Skeleton

Loading-state placeholder for content that's still in flight. One primitive bar with a slow shimmer composes every loading surface in the app — sidebar tree rows (sidebar), card grids (project-card, project-folder-view), and the home hero's "N mockups updated since yesterday" sub-line. Pairs with the fade-in wrapper that animates the skeleton → real content swap. The structural shell (sidebar logo, collapse, section labels, topbar) is never skeletoned — those elements derive from the URL + auth state, not from the per-page API.

React API — <Skeleton /> + <FadeIn />

Radix Primitives does NOT ship a skeleton or loading-state primitive. The project uses a custom compound at src/components/Skeleton/ (the bar) and src/components/FadeIn/ (the wrapper). The two are paired in usage but live in separate component folders because FadeIn is also useful outside loading transitions (modal enter, popover content).

// src/components/Skeleton/Skeleton.tsx — primitive bar with shimmer
import { Skeleton } from '@/components/Skeleton';
import { FadeIn } from '@/components/FadeIn';

// Block (default) — for thumbnails and large rectangular slots
<Skeleton width={"100%"} height={"160px"} />

// Text — height: 0.75em, radius: pill — for label-sized bars next to icons
<Skeleton width="60%" height={14} variant="text" />

// Circle — for avatars / round status dots
<Skeleton width={28} height={28} variant="circle" />

// Canonical "skeleton → content" pattern. Skeleton renders raw (no
// FadeIn — it IS the first paint). The resolved-content branch fades
// in over 360ms so the swap doesn't snap.
if (status === 'loading' || !data) {
  return <ProjectSkeleton />;
}
return (
  <FadeIn>
    <ProjectContent {...data} />
  </FadeIn>
);
  • Three variants, one component. variant="block" | "text" | "circle". Block is default and renders as a 1em-tall bar with --radius-sm corners. Text is shorter (0.75em) with a pill radius, sized for labels next to icons. Circle uses 50% radius for avatars / dots.
  • Dimensions are explicit. Callers pass width and height as strings ("60%", "120px") or numbers (14, treated as px). The component has no intrinsic sizing — pass both.
  • Compositions live with the page, not in the DS. SidebarTreeSkeleton, HomeSkeleton, and ProjectSkeleton are page-level compositions of this primitive. They mirror the post-load layout of their destination, then disappear on data arrival. New page-level surfaces author their own composed skeleton in src/components/Skeleton/<Name>Skeleton.tsx.
  • Pair every skeleton with <FadeIn> on the resolved branch. The skeleton itself renders raw (it IS the first paint — fading it in would just delay feedback). The resolved-content branch wraps in <FadeIn> so the swap is a 360ms ease-out cross-fade. Never FadeIn both sides — that double-animates.
  • Never replace the structural shell with skeleton. The sidebar's brand, collapse button, "Projects" / "No project" section labels, footer "New mockup" button, and the topbar's breadcrumbs + avatar paint from the URL + auth response. Only the variable parts (tree rows, page content cards, hero count) get skeletons.
  • Reduced motion is automatic. The shimmer animation and the FadeIn animation both zero out under @media (prefers-reduced-motion: reduce). No opt-in flag.
When to use which
Use <Skeleton /> when…Use the alternative when…
The eventual layout is predictable — you know roughly where the title, the cards, the avatar will land. Skeleton paints the silhouette so the page doesn't jump on data arrival. The eventual layout is unknown (long-running search results, paginated lists with variable counts) — render a spinner instead so you don't promise a shape you might not deliver.
The wait is > 100 ms. Below that, the skeleton-then-content swap is itself noise; render nothing and let the parent's skeleton cover the cold start. The wait is < 100 ms typical (cached refetch, optimistic UI). Either don't render a placeholder at all, or render aria-busy="true" on the existing content while it refreshes in place.
The placeholder is the first paint. Render the skeleton raw — no <FadeIn> wrap. The placeholder is part of a transition between two real states. Use <FadeIn> on the destination only.

Behavior
  • Renders a single <span aria-hidden="true"> per bar. Hidden from screen readers — the parent surface owns aria-busy="true" + aria-live="polite" so SR users hear "loading" once, not once per bar.
  • Shimmer runs continuously while mounted (1.4s loop). Stops automatically on unmount; no JS lifecycle plumbing required.
  • Under prefers-reduced-motion: reduce, the bar stays visible as a flat surface (no animated stripe). Users still see "loading"; they don't get a stripe sliding across the screen every 1.4s.
  • <FadeIn> plays once on mount (360ms ease-out, opacity + 6px slide-up). Subsequent re-renders stay at the resting state — CSS animations don't re-trigger without a fresh keyframes binding.
  • <FadeIn> is mount-only — it does NOT animate exits. If you need an exit fade, swap to a CSS-in-JS or Framer Motion solution; the current contract is intentionally one-directional because the skeleton → content transition only ever goes forward.
  • Inline display: <Skeleton variant="text" style={{ display: 'inline-block' }}> sits inside a sentence (e.g. the home hero sub-line where the surrounding date + dot are real text). Default display: block otherwise.
  • The skeleton replaces variable data, never structural chrome. If a region is derivable from the URL or already-loaded shell state, render it as real text — not as a skeleton.
