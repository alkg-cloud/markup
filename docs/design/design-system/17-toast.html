17 — Toast

Transient bottom-center pill for confirmation messages and lightweight feedback. Implemented as the custom ToastProvider + useToast hook (src/components/Toast/useToast.ts + Toast.tsx) — a context + reducer pair that holds a stack of ToastItems and auto-removes each after a duration. Renders the container with aria-live="assertive" so SR announces every message. Pills use the glass-surface standard — same recipe as the rail, toolbar, dialog scrim, and other floating overlays. For inline, persistent status (server error inside a form, rate-limit warning) reach for alert-banner instead.

React API — custom ToastProvider + useToast (not Radix)

Backed by a custom in-project implementation, not @radix-ui/react-toast. @radix-ui/react-toast is not installed; the lightweight context-+-reducer pair in src/components/Toast/ is the canonical surface. Reach for Radix Toast only if a future need surfaces (swipe-to-dismiss, action slot, queue limits) — and add it explicitly to package.json at that time.

// 1) Mount the provider once at the app root
import { ToastProvider } from '@/components/Toast/useToast';

export default function AppShell({ children }: { children: ReactNode }) {
  return <ToastProvider>{children}</ToastProvider>;     // renders the container + reducer
}

// 2) Call `show(message, duration?)` from any client component
import { useToast } from '@/components/Toast/useToast';

function CopyLinkButton() {
  const { show } = useToast();
  return (
    <button onClick={() => {
      navigator.clipboard.writeText(window.location.href);
      show('Copied to clipboard');                  // duration defaults to 3000 ms
    }}>
      Copy link
    </button>
  );
}

// 3) Custom duration (e.g. a longer confirmation)
show('Token created — copy now, you won\'t see it again', 5000);

// Public API:
//   show(message: string, duration?: number = 3000): void
// The reducer dispatches `add` then schedules `remove` after `duration`.

Notes:

  • Why not Radix Toast. The current API surface is tiny (one show() call), there's no swipe-to-dismiss requirement, no action slot, no queue policy beyond "remove after N ms". A 60-line custom hook is the right fit; adding @radix-ui/react-toast just imports machinery we don't use.
  • No variant prop yet. show() takes only message + optional duration. The .success / .danger intent classes shown above are reserved for a future show(message, { variant: 'success', duration: 4000 }) signature — flagged here so the DS sees the gap.
  • Auto-dismiss only. Toasts have no close button — that's the contract. If the user needs to dismiss, use a alert-banner with onClose instead.
  • Aria. The container is aria-live="assertive" + aria-atomic="false"; each pill carries role="status". SR announces every message as it appears.
  • Pointer-events. The fixed container is pointer-events: none; individual pills are auto. That way the bottom of the canvas remains clickable behind the stack.
State decision matrix
StateTriggerVisualAria
idle No pending toasts. Default state. Container exists (aria-live region) but contains no children. aria-live="assertive" on the container — quiet until a child appears.
entering Reducer dispatch add after show(message). Pill mounted invisible; immediately animates opacity 0 → 1, translateY 8 → 0, scale 0.95 → 1 over --motion-fast with --ease-spring. Pill carries role="status". SR reads message as it enters.
visible Animation complete. Pill held on screen for duration ms (default 3000). Pill at full opacity, on top of any older pills (newest-bottom-front). —
exiting setTimeout fires remove dispatch after duration. Opacity 1 → 0 over 250 ms with --ease-exit. Older pills slide down to fill the slot. SR ignores the disappearance.
success (custom) Not yet wired — reserved for the future show(msg, { variant: 'success' }) signature. Same glass surface, leading icon coloured --success. Same as visible. SR announcement unchanged (the icon is decorative).
danger (custom) Not yet wired — reserved for the future show(msg, { variant: 'danger' }) signature. Same glass surface, leading icon coloured --danger. Same as visible. Errors that need user action should use alert-banner, not a toast.
"success" and "danger" are future states — the current useToast hook only ships the neutral pill. Adding them later means extending show() with an options object (no breaking change).
When to use which
PrimitiveUse forBacking
toast Transient confirmation ("Copied", "Saved", "Moved to X"). Auto-dismisses. No action. Custom — ToastProvider + useToast
alert-banner Persistent inline status (rate-limit warning, server error in a form, success of a long-running task). May have a close button or an action. See 27-alert-banner.html
dialog + useConfirm Destructive confirmations or blocking flows that demand a user choice. Toasts must never be used for "are you sure?". See 14-dialog.html
input-field error Per-field validation message inside a form body. Stays attached to the field. See 28-input-field.html
Rule of thumb: if the user must act on it, it's not a toast. Toasts confirm what just happened — they don't ask for input or stay around.

Behavior
  • Imperative show. Any client component reads useToast() and calls show(message, duration?). No render-time wiring required.
  • Stack ordering. Newest pill at the bottom-front via flex-direction: column-reverse. Older pills shift up — they don't reorder, they just move.
  • Auto-dismiss. setTimeout fires remove dispatch after duration (default 3000 ms). No manual close — that's intentional.
  • No queue cap. Multiple show() calls in quick succession stack indefinitely. If this proves noisy in practice, a future cap (e.g. max 3 visible) lives in the reducer, not the consumer.
  • Aria live region. Container is aria-live="assertive" + aria-atomic="false"; each pill is role="status". SR announces every message as it appears.
  • Pointer-events: none on container. The fixed container doesn't block clicks behind it; individual pills are pointer-events: auto so they can be clicked (currently no-op — future variants may add an action button).
  • Position lock. Container is always bottom-centre, fixed. No portal — the container lives wherever ToastProvider renders (at the app root).
  • Reduced motion. Under prefers-reduced-motion: reduce, the enter / exit animations collapse to a 0 ms opacity flip via the token override at the top of the file.
