React API — composes RadixDialog + Form.Root + InputField + segmented (custom)

Tier-3 means: compose three Radix-backed primitives (RadixDialog, Form.Root, InputField), then add one custom compound the project owns (segmented). Verbatim shape from src/app/(app)/settings/invites/InvitesClient.tsx — abridged for clarity.

import * as Form from '@radix-ui/react-form';
import { RadixDialog } from '@/components/Dialog/RadixDialog';        // thin Radix Dialog wrapper
import { InputField } from '@/components/InputField';             // see 28-input-field

<RadixDialog.Root open={open} onOpenChange={handleOpenChange}>
  <RadixDialog.Portal>
    <RadixDialog.Overlay />
    <RadixDialog.Content aria-describedby={undefined}>
      <RadixDialog.Title>Create invite</RadixDialog.Title>

      <Form.Root className={styles.dialogForm} onSubmit={handleFormSubmit}>
        <InputField.Root name="email" data-state={emailError ? 'error' : undefined}>
          <InputField.Label>Bind to email (optional)</InputField.Label>
          <InputField.Control asChild>                              // Slot — native <input>
            <input type="text" value={email} onChange={…} placeholder="name@example.com" />
          </InputField.Control>
          {emailError
            ? <InputField.Message forceMatch>{emailError}</InputField.Message>
            : <InputField.Help>If set, signup requires this exact email.</InputField.Help>}
        </InputField.Root>

        /* Segmented role + expiry — custom compound (no Radix primitive). */
        <div className={styles.dialogField}>
          <span className={styles.dialogLabel}>Role</span>
          <div className={`${styles.segmented} ${styles.cols2}`} role="radiogroup">
            {['member', 'admin'].map((opt) => (
              <button key={opt} type="button"
                role="radio" aria-checked={role === opt}
                className={`${styles.seg} ${role === opt ? styles.segActive : ''}`}
                onClick={() => setRole(opt)}>…</button>
            ))}
          </div>
        </div>

        <div className={styles.dialogActions}>
          <RadixDialog.Close asChild>                                // Slot — Cancel
            <button type="button" className={styles.dialogBtnSecondary} disabled={busy}>Cancel</button>
          </RadixDialog.Close>
          <Form.Submit asChild>                                      // Slot — Create
            <button type="submit" className={styles.dialogBtnAccent} disabled={busy}>
              {busy ? 'Creating…' : 'Create invite'}
            </button>
          </Form.Submit>
        </div>
      </Form.Root>
    </RadixDialog.Content>
  </RadixDialog.Portal>
</RadixDialog.Root>

Notes:

  • Three Radix primitives, one custom. RadixDialog (= @radix-ui/react-dialog wrapper, focus trap + scroll lock + portal) + Form.Root (sync validation + ValidityState) + InputField (project's compound — see 28-input-field). Segmented control is custom — Radix Primitives ships no segmented/toggle-group with native radio semantics that fits this look, so the project keeps a simple radio-flavored button row.
  • Error state on the wrapper, not the input. The InputField.Root gets data-state="error"; CSS scopes the danger ring + shake to children. Same pattern as 28-input-field. The error message lives in InputField.Message with forceMatch — we drive validity from the server response, not Radix's ValidityState.
  • Submit + Cancel via asChild. Both Radix wrappers expose asChild so the consumer renders the native <button> with the project's dialog-btn styling — no Radix "Button" primitive exists (per 13-buttons).
  • No autoFocus. Radix Dialog auto-focuses the first focusable child of Content on open. Adding autoFocus also trips Biome's useFocusableInteractive.
  • Close X. RadixDialog.Content supports showCloseButton (see 14-dialog § close-btn). This dialog is a single-field form; by the DS guideline for single-field forms the X is optional — the Cancel button in the action row is sufficient.
  • Submission flow. handleFormSubmit calls onSubmit (parent InvitesClient) which POST /api/invites, captures the plaintext token from the response, copies it to the clipboard, prepends the row, and closes the dialog. Errors map back to emailError for the input.
State decision matrix
StateTriggerVisualAria
idle Dialog opens. Default selection: Member + 7d. Form rendered, email field focused. aria-modal via Radix; RadixDialog.Title labels.
error (custom) Server returns 4xx with field error, or sync validation fails. InputField.Root gets data-state="error" → danger ring + shake. Help text swapped for InputField.Message. aria-invalid + aria-describedby via InputField.
busy (custom) busy={true} while fetch in flight. All inputs disabled; CTA label → "Creating…"; close button blocked. Native disabled announces to SR.
closing User clicks Cancel, ✕, presses Esc, or clicks the overlay. Radix Dialog exit animation (data-state="closed"). Focus returns to the trigger element.
error and busy are custom DS states — neither Radix Dialog nor Radix Form models them with built-in attributes (Radix Form has [data-invalid] but the project surfaces server errors via InputField's data-state wrapper instead).

Behavior
  • Focus trap + scroll lock. Provided by @radix-ui/react-dialog via RadixDialog.Content. Tab cycles inside the dialog; the body underneath is non-interactive until close.
  • Initial focus. Radix auto-focuses the first tabbable child of Content — the email InputField. Never set autoFocus manually (Biome forbids it).
  • Dismiss. Esc, click on overlay, click ✕, or click Cancel — all close. onOpenChange(false) drives the close. While busy, the close button is disabled but Esc still works (Radix default) — that's intentional so a stuck request doesn't trap the user.
  • Submission. Submit handler calls the parent onSubmit({email, role, expiry}) which performs POST /api/invites. On 2xx the parent prepends the new row, copies the plaintext token to clipboard, and closes the dialog. On 4xx the parent surfaces the error string back through setEmailError.
  • Auto-copy token. The plaintext token is only ever shown once (the server stores SHA-256). Auto-copy + toast confirmation is the single user-visible reveal — same posture as agent-tokens.
  • Defaults. Role=Member, Expiry=7d, Email=empty. These match the most-common path: ad-hoc teammate invite, week-long link, no email binding.
  • Reduced motion. Both the enter animation and the error-shake collapse to 0 ms via the token override.
