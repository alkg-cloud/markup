15 — Popover

Anchored overlay primitive — every kebab menu, status toggle, version row, picker, and avatar dropdown. Two implementations coexist: src/lib/popover/usePopover.ts wraps the native HTML popover API (popover="auto" + popovertarget) and is the default for in-tree kebabs (project tree, annotation card, comment, version row, topbar avatar); FolderPicker uses @radix-ui/react-popover when it needs trigger-width matching (--radix-popover-trigger-width) and React-controlled state. Both paint in the browser top-layer / portal, escaping every overflow ancestor. Shares the glass-surface standard with dialog, tooltip, and rail/toolbar: --surface-glass-bg, blur 16 px / saturate 140%, --border, --radius-sm, --shadow-popover.

React API — usePopover (default) + Radix Popover (FolderPicker)

Two implementations live in the project. Default kebabs use the in-project usePopover hook (src/lib/popover/usePopover.ts) which wraps the native HTML popover API. Folder pickers and other trigger-width-matching anchors use @radix-ui/react-popover for its CSS variable --radix-popover-trigger-width and React-controlled open state.

// 1) Default kebab menu — usePopover wraps native popover="auto"
import { usePopover } from '@/lib/popover/usePopover';

function KebabMenu() {
  const { triggerRef, popoverProps, close } =
    usePopover<HTMLButtonElement, HTMLDivElement>('right');

  return (
    <>
      <button ref={triggerRef} className={styles.kebab}
              aria-haspopup="menu" />          // popovertarget wired by triggerProps
      <div {...popoverProps} className={styles.popover} role="menu">
        <button className={styles.item}
                onClick={() => { close(); doEdit(); }}>Edit</button>
        <div className={styles.divider} />
        <button className={`${styles.item} ${styles.danger}`}
                onClick={() => { close(); doDelete(); }}>Delete</button>
      </div>
    </>
  );
}

// 2) Trigger-width-matching popover — @radix-ui/react-popover (FolderPicker)
import * as Popover from '@radix-ui/react-popover';

<Popover.Root open={open} onOpenChange={setOpen}>
  <Popover.Trigger asChild>                          // Slot — caller owns the button
    <button className={styles.trigger}>…</button>     // gets [data-state="open"]
  </Popover.Trigger>
  <Popover.Portal>
    <Popover.Content
       align="start" sideOffset={6}
       className={styles.popover}
       style={{ width: 'var(--radix-popover-trigger-width)' }}>   // match trigger width
      …
    </Popover.Content>
  </Popover.Portal>
</Popover.Root>

Notes:

  • usePopover is the canonical hook. Reach for it for every kebab, status toggle, version row, comment menu, and topbar avatar — anything that's a simple "anchored list of actions".
  • @radix-ui/react-popover is reserved for two niches: (a) trigger-width matching via --radix-popover-trigger-width (FolderPicker), (b) when you need React-controlled open tied to outer state (form fields whose popover snaps shut on submit).
  • Both render in the top-layer / portal — never trapped in transformed ancestors. The CSS must set position: fixed; inset: auto; margin: 0 on the popover so the JS-written top / left win over the browser-default centring.
  • Menu items are native <button> — Radix has no Button primitive. Pair each onClick with close() so the action fires and the popover closes in the same gesture.
  • role="menu" on the container + role="menuitem" on items is correct for kebabs; role="radiogroup" applies to the status toggle group at the top of annotation-card kebab.
  • Light-dismiss (click-outside) and ESC are browser-native for usePopover (via popover="auto") and Radix-driven for Popover.Root. Both arrive without extra wiring.
State decision matrix
StateTriggerVisualAria
closed Default. Trigger has popovertarget set but the popover is hidden. Nothing in the top-layer. Trigger: aria-haspopup="menu", aria-expanded="false".
open Click trigger or invoke showPopover() programmatically (Radix: open={true}). Popover paints in the top-layer at calculated position. Glass surface, 16 px blur, --shadow-popover. Trigger gets .active / [data-state="open"] for accent border. Trigger: aria-expanded="true". Container: role="menu". First menuitem gets focus.
flipped (auto) Default-below placement would overflow the viewport gutter (4 px). Position calculator switches to above-the-trigger. No visual change to surface itself — just the top value. Same as open. Reading order unchanged.
nested-open A second popover opens while another is already open (only valid for non-sibling triggers — e.g. tooltip inside an open popover). Both surfaces visible. Top-layer paints them in DOM order — newer-on-top. Each carries its own role; focus is in the most recently opened.
disabled Trigger button has disabled attribute (e.g. FolderPicker with projectId === null). Trigger: 0.4 opacity, cursor:not-allowed. Click does nothing — popover never opens. Trigger: aria-disabled="true" (native via disabled).
"flipped" and "nested-open" are derived states — the consumer doesn't toggle them. The position calculator and the top-layer rules handle both automatically.
When to use which
PrimitiveUse forBacking
usePopover Default kebab menus, status toggles, version row menus, comment menus, topbar avatar — anything that's a simple anchored action list. Native popover="auto" + custom positioner — src/lib/popover/usePopover.ts
@radix-ui/react-popover Pickers needing trigger-width matching (--radix-popover-trigger-width) or React-controlled open state tied to outer logic. Currently: FolderPicker. @radix-ui/react-popover (installed)
tooltip Non-interactive hover hints. Smaller, no interactive items, opens on hover/focus only. See 16-tooltip.html
dialog Modal flows that own the user's attention — forms, confirmations. Centred, scrim, focus trap. See 14-dialog.html
Rule of thumb: if the surface anchors to a trigger and the user can dismiss by clicking outside, it's a popover. If it blocks the page until the user makes a choice, it's a dialog. If it appears on hover without interactive content, it's a tooltip.

Behavior
  • Open / close. usePopover: browser owns it (popover="auto" + popovertarget). Radix: open + onOpenChange on Popover.Root.
  • Light-dismiss. Click outside the popover closes it (browser-native for popover="auto", Radix-default for Popover.Root).
  • Escape closes. Both implementations dismiss on Escape and return focus to the trigger.
  • Single-popover-auto. popover="auto" auto-closes any other open popover="auto" element when a new one opens. Use popover="manual" only when stacking is required (none of our surfaces need it).
  • Positioning. Default "below the trigger, align left/center/right"; flips to above when the viewport gutter (4 px) would be violated; clamps horizontally to stay on-screen.
  • Top-layer paint. Both implementations escape every overflow ancestor and stacking context — the popover is never clipped by a parent overflow: hidden or stacked beneath a sibling.
  • Trigger active state. usePopover doesn't set a class — consumer toggles .active if visual feedback is needed. Radix sets [data-state="open"] automatically; CSS can hook into it.
  • Reduced motion. Both implementations respect prefers-reduced-motion: reduce — Radix's enter animation collapses to a 0 ms opacity flip; usePopover has no enter animation by default.
