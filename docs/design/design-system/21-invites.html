21 — Invites

Admin-only page at /settings/invites. Generates invite links for new teammates. Three canonical list states (populated, first-visit onboarding, post-activity terse) plus the bulk-actions popover anchored to the header kebab. Opens 22-new-invite-dialog from "New Invite"; destructive bulk actions go through useConfirm. The first-visit → post-activity transition fires on either of two events (admin creates the first invite OR dismisses the onboarding card) and is one-way (persisted in localStorage). Implemented by src/app/(app)/settings/invites/InvitesClient.tsx.

First-visit shows the heavy onboarding card (steps + secondary CTA) — for admins who've never opened the page. Post-activity renders after the admin either (a) creates an invite or (b) explicitly dismisses the onboarding card. The flag (ONBOARDING_KEY = 'markup.invites.onboarding-dismissed') is stored in localStorage and the transition is one-way — once an admin has seen the terse state, the heavy card never returns.
C — Bulk-actions popover (anchored to header kebab)
Both items render with the live count chip of affected invites — when the count is zero the item is disabled (right preview), not hidden, so the menu shape stays consistent regardless of list state. Each action opens a useConfirm dialog before firing. The popover anchors to the kebab via the project's usePopover() hook — not a Radix DropdownMenu, see React API below.
React API — page composition (no Radix primitive for the page itself)

The page itself is a custom client component — no Radix primitive backs the row/list/header anatomy. It composes the project's AppMain shell, opens 22-new-invite-dialog (which IS Radix-backed: RadixDialog + Form.Root + InputField), uses the project's useConfirm compound (which IS Radix-backed: @radix-ui/react-alert-dialog), and anchors the bulk-actions menu via the project's usePopover hook (custom — see notes). Source: src/app/(app)/settings/invites/InvitesClient.tsx.

'use client';
import * as Form from '@radix-ui/react-form';
import { AppMain } from '@/components/AppMain/AppMain';
import { useConfirm } from '@/components/ConfirmDialog';       // AlertDialog-backed
import { RadixDialog } from '@/components/Dialog/RadixDialog';  // Dialog-backed
import { InputField } from '@/components/InputField';
import { useToast } from '@/components/Toast/Toast';
import { usePopover } from '@/lib/popover/usePopover';             // custom — see notes

export function InvitesClient({ initialInvites }) {
  const [invites, setInvites] = useState(initialInvites);
  const [onboardingDismissed, setOnboardingDismissed] = useState(false);
  const [plaintextByInviteId, setPlaintextByInviteId] = useState({});      // session-only
  const [showDialog, setShowDialog] = useState(false);
  const bulkMenu = usePopover<HTMLButtonElement, HTMLDivElement>('right');
  const { confirm, dialog: confirmDialog } = useConfirm();
  const toast = useToast();

  const total = invites.length;
  const showOnboarding = total === 0 && !onboardingDismissed;
  const showTerse      = total === 0 &&  onboardingDismissed;

  return (
    <AppMain variant="scroll">
      <h1>Invites</h1>
      <p>Generate invite links for new teammates.</p>

      <div className={styles.listHeader}>
        <span className={styles.count}>{total} invites</span>
        <div className={styles.headerActions}>
          <button type="button" className={styles.btnAction} onClick={() => setShowDialog(true)}>
            <VscAdd /> New Invite
          </button>
          <button type="button" ref={bulkMenu.triggerRef} {...bulkMenu.triggerProps} className={styles.kebabBtn}>
            <VscKebabVertical />
          </button>
        </div>
      </div>

      {showOnboarding ? <EmptyOnboarding onDismiss={…} onCreate={…} /> :
       showTerse      ? <EmptyTerse /> :
                        invites.map((row) => (
                          <InviteRow key={row.id} row={row}
                            plaintext={plaintextByInviteId[row.id]}                // session-only reveal
                            onRevoke={(r) => confirm({...}).then(...)}             // AlertDialog
                            onRemove={(r) => confirm({...}).then(...)}             // AlertDialog
                          />
                        ))}

      {bulkMenu.isOpen && (
        <div ref={bulkMenu.contentRef} {...bulkMenu.contentProps} className={styles.kebabMenu} role="menu">…</div>
      )}

      <NewInviteDialog open={showDialog} onClose={() => setShowDialog(false)} onSubmit={…} />
      {confirmDialog}                                                              // Portaled AlertDialog
    </AppMain>
  );
}

Notes:

  • No Radix primitive for the page itself. AppMain is the project's scroll-container shell; the list / row / header are plain HTML + CSS Modules. The only Radix-backed surfaces invoked from this page are: NewInviteDialog (Dialog + Form), useConfirm (AlertDialog), and the buttons-via-asChild pattern inside those (see 13-buttons).
  • Bulk-actions popover uses usePopover, not Radix. The project's usePopover in src/lib/popover/usePopover.ts is a small custom hook for anchored popovers — it predates the decision to standardize on Radix. Migration to @radix-ui/react-popover or react-dropdown-menu is parked in docs/future-features.md; the current behavior (click anchor → toggle, click outside → close, Esc → close) is correct.
  • Destructive actions go through useConfirm. Revoke single row, remove single row, "Revoke all open", "Clear all history" — each opens an AlertDialog with action-specific copy. confirmDialog must be rendered in the JSX tree once so the portaled dialog mounts.
  • Plaintext token reveal is session-local. The server returns the plaintext token only in the response to POST /api/invites. The client stores it in plaintextByInviteId state and shows it on the just-created row. On reload, the map is empty and the row renders without the reveal — same security posture as agent-tokens.
  • Onboarding flag. Read from localStorage[ONBOARDING_KEY] on mount; written when the admin dismisses or creates their first invite. Once true, never reverts.
  • Admin-only route. Behind /settings/invites; the middleware redirects non-admins to /. The client also re-checks via useRequireAuth({ role: 'admin' }) as a defensive double-check.

Behavior
  • Admin-only. Behind /settings/invites. Middleware redirects non-admins; client also re-checks. Members never see this page.
  • One-time plaintext reveal. Token is returned only by POST /api/invites. Stored in component state (plaintextByInviteId) for the session; auto-copied to clipboard; surfaced as a token-reveal strip on the just-created row. Page reload = strip gone forever. Same posture as AgentToken.
  • Row actions vary by state. Active rows (unused) show Copy + Revoke; terminal rows (used/expired/revoked/disabled) show only "Remove from list" on hover — there's nothing to copy or revoke once the invite is dead.
  • Revoke vs Remove. Revoke flips an unused invite to terminal (server-side, undoable by an admin? No — design choice: revoke is irreversible). Remove deletes the row from the admin's history view but never resurrects the underlying invite — same destructive semantic, different scope.
  • Bulk actions. "Revoke all open invites" calls POST /api/invites/revoke-all; "Clear all history" calls POST /api/invites/history/clear. Both go through useConfirm first. Zero-count items render disabled (not hidden) so the menu shape is stable.
  • Onboarding card transition. First-visit → terse fires on either (a) admin creates the first invite OR (b) explicit dismiss of the onboarding card. Persisted in localStorage[ONBOARDING_KEY]. One-way — once gone, never returns.
  • Empty after activity. "Revoke all" then "Clear all" lands the admin in the post-activity terse empty (NOT the first-visit onboarding) — because onboardingDismissed is already true at that point.
  • No optimistic deletes. Each destructive action waits for the API response before mutating the list — feedback comes via the toast. (The list is small enough that latency isn't a UX concern.)
  • Reduced motion. No bespoke animations beyond color/border/opacity transitions, which collapse to 0 ms via the token override.
