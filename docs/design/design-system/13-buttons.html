13 — Buttons

Every clickable surface in the app — page-level CTAs, table actions, dialog footers, icon triggers — composes the same base button recipe. Variants are semantic hues on a single shape (Action / Secondary / Ghost / Danger); sizes are data-size attributes; loading and disabled are state attributes. A second pill-shaped family lives inside dialog footers. Both are native <button> elements styled via CSS modules.

Radix Primitives ships no Button primitive — and that is deliberate. Radix avoids the "kitchen-sink Button" trap because buttons are pure CSS + native semantics with no a11y problem worth solving via abstraction. The project follows suit: every button is a native <button type="button"> with a CSS module class. When a button needs to behave like a Radix child (e.g. RadixDialog.Close, Form.Submit, AlertDialog.Action), wrap it with the parent's asChild prop — Radix's @radix-ui/react-slot merges the props through. See ConfirmDialog.tsx and NewProjectDialog.tsx for the canonical pattern.
All variants — default size (32 px)
All variants share the same .btn rule. Variant via [data-variant="…"]. 32 px tall, 13 px text, 600 weight, --radius-sm corner.
Small size (26 px)
Reach for [data-size="sm"] in tight contexts — inline-table actions, kebab-popover menus. 26 px tall, 12 px text. Don't drop below 26 px; use icon-buttons instead.
Icon buttons
Icon-only buttons MUST carry aria-label (Biome a11y rule enforces). Active state via [data-state="active"]. Mini size (22 px) for embedded contexts — close glyphs in dialog headers, kebab menus.

Dialog buttons — pill-shaped family
A separate, pill-radius family used only inside dialog footers. Verbatim from NewProjectDialog.module.css and RadixDialog.module.css. 30 px tall, --radius-pill corner, 12.5 px text, 700 weight. The two button families exist because page CTAs and dialog action buttons sit in very different visual contexts — page buttons live among titles and cards (rectangular rhythm), dialog buttons live below a single content blob and benefit from the rounder gesture.
React API — native <button> + asChild slot

Radix Primitives ships no Button primitive. This is intentional: a button is just <button type="button"> + CSS. The project follows the same convention. When a button needs to participate in a Radix compound (close the dialog, submit the form, confirm a destructive action), wrap it via the compound's asChild prop — @radix-ui/react-slot merges onClick, aria-*, refs through to the native element.

// 1) Standalone button — no Radix involved
<button
  type="button"
  className={styles.btn}
  data-variant="action"
  onClick={handleClick}
>
  <VscAdd /> New Project
</button>

// 2) Inside Radix Dialog — Slot via asChild keeps the native element
import { RadixDialog } from '@/components/Dialog/RadixDialog';
import * as Form from '@radix-ui/react-form';

<RadixDialog.Close asChild>                                // Slot — caller owns the <button>
  <button type="button" className={styles.btnSecondary}>Cancel</button>
</RadixDialog.Close>

<Form.Submit asChild>
  <button type="submit" className={styles.btnAccent}>Create</button>
</Form.Submit>

// 3) Inside ConfirmDialog — AlertDialog.Action + AlertDialog.Cancel
import * as AlertDialog from '@radix-ui/react-alert-dialog';

<AlertDialog.Cancel asChild>
  <button type="button" className={styles.btnSecondary}>Cancel</button>
</AlertDialog.Cancel>
<AlertDialog.Action asChild>
  <button type="button" className={styles.btnDanger}>Delete</button>
</AlertDialog.Action>

Notes:

  • Always specify type="button" unless the button is genuinely a form submit (then type="submit"). Without it, browsers default to type="submit" inside <form> and your button accidentally fires the form's submit handler.
  • Never autoFocus. Biome's a11y rule (useFocusableInteractive) forbids it. Radix Dialog focuses the first focusable child of Content automatically on open — that's the right pattern.
  • Icon-only buttons require aria-label. Biome rule (useButtonType + useValidAriaProps). The label is the action name in the user's language ("Delete folder", not "Trash icon").
  • Don't reinvent <a> as a button. If clicking navigates the URL, use <Link> from next/link styled with a button class. Don't put onClick={() => router.push(…)} on a <button> — middle-click and ⌘-click won't open a new tab.
  • Variant class vs data-variant. Existing code uses class composition (styles.btn + styles.btnSecondary) — that's fine and the production CSS modules expose it that way. New code can adopt the data-variant pattern shown here for parity with the alert-banner / input-field migration.
State decision matrix
StateTriggerVisualAria
rest Default. Variant base colors. cursor:pointer. Native button semantics.
hover Pointer over. Bg + border + color shift per variant (see :hover:not(:disabled) rules). —
focused Keyboard focus (:focus-visible). Standard --focus-ring: 2 px --bg + 4 px --accent halo. Native focus ring.
disabled HTML disabled attribute. Opacity 0.4, cursor:not-allowed, no hover effects. Native button semantics announce "disabled" to SR.
loading (custom) Consumer sets [data-state="loading"] + swaps icon for .spinner. Opacity 0.7, cursor:wait. Spinner spins at 0.6 s/turn. Pair with disabled while in flight so SR doesn't announce a still-active button.
active (icon-btn only) Consumer sets [data-state="active"] — a toggle is on (eg comments panel open). --accent-overlay-soft bg, --accent-bright color. aria-pressed="true" on the button if it's a toggle.
"loading" and "active" are custom DS states — neither Radix nor HTML model them. Consumers manage the attribute; the CSS handles the visual.
When to use which
VariantUse forDon't use for
action The primary CTA in a page or section header — "New Project", "Upload", "Invite". One per scope. Anything destructive (use danger).
secondary The non-primary action in a pair — "Edit" next to an "Open" action, "New Folder" next to "New Mockup". Standalone toolbar buttons (use ghost).
ghost Inline cancel, tertiary toolbar actions, dismiss controls. The primary CTA — it's too quiet to be a default.
danger Destructive confirmations: delete, revoke, discard. Pair with useConfirm. "Cancel" in a non-destructive flow.
icon-btn Inline single-purpose action where the icon alone is unambiguous (delete row, copy chip, close modal). Multi-step primary actions — they need a label.
dialog-btn Footer actions inside a dialog. Pill radius, more weight. Anywhere outside a dialog footer.
Rule of thumb: one action per scope. Multiple actions = action + secondary + (optional) ghost Cancel. Confirmations involving data loss = danger via useConfirm.

Behavior
  • Native semantics. All buttons are <button type="button"> (or type="submit" inside Form.Submit). No role="button" on divs, ever.
  • asChild for Radix compounds. Inside RadixDialog.Close, Form.Submit, AlertDialog.Action, AlertDialog.Cancel: always use asChild + a native <button>. The Slot merges props through.
  • Disabled + loading pair. Set both disabled AND [data-state="loading"] while a request is in flight. Disabled blocks the click; data-state drives the spinner + cursor.
  • Loading swaps the icon, not the label. The leading-icon slot becomes <span class="spinner" />; the label changes to verb-progressive ("Creating…"). Don't blank the label — that drops affordance.
  • Focus ring is global. :focus-visible on any element triggers the canonical 2 px --bg + 4 px --accent halo from the base reset. Don't override it on buttons.
  • Icon-only requires aria-label. Without it, screen readers announce nothing or the SVG title. Use action-name labels in the user's language.
  • Don't combine action + danger on the same element. A primary destructive action means danger alone — the variant is the priority signal.
  • Reduced motion. Spinner respects prefers-reduced-motion: reduce via the token override; color/border transitions collapse to 0 ms.
