Input field

Single-line text input with label, optional help line, and stateful variants. Error covers field-level validation (required, format, length, duplicate); success covers affirmative confirmation (e.g., async availability check passed). Non-field failures go to the alert-banner instead.

React API — composes with Radix Form

Radix Primitives ships @radix-ui/react-form with Form.Root, Form.Field, Form.Label, Form.Control, Form.Message, Form.Submit. It manages validity via the browser's ConstraintValidation API and emits [data-invalid] / [data-valid] on Form.Field when validation state changes. Our DS keys off both data-invalid (Radix-driven sync validation) and data-state="error" | "success" (consumer-driven for async/server validation).

// InputField.tsx — composes Radix Form primitives
import * as Form from '@radix-ui/react-form';

<Form.Root onSubmit={handleSubmit}>
  <Form.Field name="name" className="input-field">        // gets [data-invalid] / [data-valid]
    <Form.Label>Name</Form.Label>
    <Form.Control asChild>                            // Slot — keep native <input>
      <input type="text" required pattern="[a-z0-9-]+" />
    </Form.Control>
    <Form.Message match="valueMissing" className="help">
      A name is required.
    </Form.Message>
    <Form.Message match="patternMismatch" className="help">
      Use lowercase letters, numbers, or hyphens only.
    </Form.Message>
    <Form.Message match={(value) => asyncCheckTaken(value)} className="help">
      A mockup with that name already exists in this folder.
    </Form.Message>
  </Form.Field>

Notes:

  • Form.Field auto-stamps data-invalid when the embedded input's ValidityState fails any matcher. Our CSS picks this up alongside [data-state="error"] so both work.
  • Form.Field also stamps data-valid when all matchers pass, but we treat that as resting — no visual. Showing a green "valid" state on every untouched input is noise.
  • For affirmative success (async name-available, code redeemed, etc.) set data-state="success" on the field manually — Radix has no built-in for this.
  • Form.Control asChild wraps a native <input> (no Radix Input primitive — same convention as the buttons in ConfirmDialog).
  • Form.Message match accepts built-in matchers (valueMissing, typeMismatch, patternMismatch, tooShort, tooLong, rangeUnderflow, rangeOverflow, stepMismatch, badInput) or a custom predicate function for async/server validation.
  • For server-side validation results from a fetch response, set data-state="error" + render the message in .help directly — Radix Form doesn't round-trip server errors automatically.

Behavior
  • Error state set by Radix Form (sync, via [data-invalid]) or by us (async/server, via [data-state="error"]). Both paths render the same visual.
  • Setting [data-state="error"] auto-triggers the shake — no JS needed. Re-apply to replay.
  • Success state ONLY via [data-state="success"] — there is no Radix primitive for this. Trigger after async confirmation (debounced).
  • .help slot is the same DOM position for resting / error / success — no layout shift when state changes.
  • Error focused state keeps danger border (doesn't fall back to accent) so the error stays legible while editing.
  • Clear state by removing data-state when the user types — error returns to default; success persists or clears on the consumer's logic.
