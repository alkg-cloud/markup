23 — Invite Signup

Public landing for an invite token. Route /invite/[token] — lives OUTSIDE src/app/(app)/, so the auth shell (Topbar, IdentityContext, useRequireAuth) never runs. This page must work for unauthenticated visitors; that is the whole point. It also redirects already-signed-in users to /. Pairs with 22-new-invite-dialog (the admin side of the same flow) and reuses the wordmark + striped-frame composition from src/app/setup/. Implemented by src/app/invite/[token]/page.tsx.

Open and email-bound invites are visually indistinguishable at rest. The only signal of a bound invite is the helper text under the email field ("Use the email your admin used to invite you"). The hint is rendered only when state.boundEmail === true — see page.tsx ~L408. This protects against email enumeration: an attacker who guesses a token can't tell from the rendered page whether a specific email is required.

All four terminal states share the warning-card template — only the title + body strings differ. TERMINAL_COPY in page.tsx maps each reason to its {title, body}. There is also a fifth fallback (unknown) used when the /state endpoint errors. The form is never rendered when the invite is unusable — there is no "try anyway" path.
React API — public route, no auth shell, no Radix primitive

This page deliberately uses neither RadixDialog nor Form.Root. It's a full-screen public route, not a dialog inside an app, so the project follows the same pattern as src/app/setup/: native <form> + inline styles + one tiny <style> block for pseudo-class state. The page is its own client component; data fetching is fetch against the route handlers. No Radix primitive backs it — the form is <form onSubmit={…}> with three <input>s.

// src/app/invite/[token]/page.tsx — abridged
'use client';
import { useRouter } from 'next/navigation';
import { useEffect, useState } from 'react';

type InviteState =
  | { usable: true; boundEmail: boolean }
  | { usable: false; reason: 'expired' | 'used' | 'revoked' | 'disabled' | 'unknown' };

export default function InviteSignupPage({ params }) {
  const router = useRouter();
  const [state, setState] = useState<InviteState | null>(null);
  const [token, setToken] = useState<string | null>(null);
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [name, setName] = useState('');
  const [error, setError] = useState<string | null>(null);
  const [busy, setBusy] = useState(false);

  useEffect(() => {
    (async () => {
      const { token } = await params;
      setToken(token);
      // Signed-in users skip the landing entirely.
      const me = await fetch('/api/auth/me');
      if (me.ok) { router.replace('/'); return; }
      const r = await fetch(`/api/invites/${token}/state`);
      setState(r.ok ? await r.json() : { usable: false, reason: 'unknown' });
    })();
  }, [params, router]);

  async function onSubmit(e) {
    e.preventDefault();
    if (!token || !state?.usable) return;
    setBusy(true); setError(null);
    const res = await fetch(`/api/invites/${token}/redeem`, {
      method: 'POST',
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({ email, password, name }),
    });
    setBusy(false);
    if (res.status === 201) { router.replace('/'); return; }
    // Map error codes onto inline error string (see Behavior).
    …
  }

  if (!state) return null;                          // pre-bootstrap, avoid flash
  if (!state.usable) return <WarningCard reason={state.reason} />;
  return <form onSubmit={onSubmit}>…</form>;
}

Notes:

  • Public route. Lives at src/app/invite/[token]/page.tsx — NOT under (app), so the auth shell never wraps it. middleware.ts explicitly allowlists /invite/* for unauthenticated traffic.
  • Two API endpoints. GET /api/invites/[token]/state returns usability + binding info; POST /api/invites/[token]/redeem creates the user + session and returns 201. Both are public (no Bearer auth) — the token is the auth.
  • Pre-bootstrap render = null. Returning null until /state resolves prevents a flash of the wrong template (e.g. showing the form for a revoked invite for 200 ms before the warning-card replaces it).
  • Signed-in pre-empt. If /api/auth/me returns 200, the page silently redirects to /. The product decision is that landing on an invite while already signed in is meaningless — accept-into-current-org is a different surface (not built yet; see docs/future-features.md).
  • No global toast bus. The setup-page lineage doesn't have access to useToast. Errors render inline (.signup-error block).
  • Inline styles + tiny <style> block. Pseudo-class states (:focus-visible, :hover, :disabled) live in LOCAL_STYLE; everything else is a React.CSSProperties object. This matches the src/app/setup/ precedent and lets the file ship without a CSS module.

Behavior
  • Public — no auth required. The page is mounted by src/app/invite/[token]/page.tsx outside the (app) group. Middleware whitelists /invite/*. No useRequireAuth / no identify() server-side.
  • Anti-enumeration. Open and email-bound invites are pixel-identical at rest — the helper text under the email field is the only hint, and the mismatch error only fires on submit. This prevents attackers from probing whether a specific email "belongs" to a token.
  • Signed-in fallthrough. If /api/auth/me returns 200, the page silently redirects to /. No toast (no toast bus available here).
  • Submit lifecycle. POST /api/invites/[token]/redeem creates the user + sets the session cookie on a 201, after which router.replace('/') takes the new user to the project list. 4xx maps to an inline error.
  • Race recovery. If the invite transitions to terminal between /state and /redeem (admin revokes mid-signup), the server returns 'invite_unusable'; the page refetches /state and swaps to the warning-card template — no jarring "submit just failed" with no context.
  • Rate limit. A 429 from /redeem renders "Too many attempts. Try again later." but does NOT disable the form — the user may want to try a different email shortly. The server's disabled protection kicks in via the regular invite state transition (terminal-disabled card).
  • No autoFocus. Same Biome rule that applies to dialogs. The browser's default focus order (email → password → name → submit) is the right one here.
  • Reduced motion. No bespoke animations — only the input focus-ring transition, which collapses to 0 ms via the token override.
