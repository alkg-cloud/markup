11 — Mockup View

The full mockup viewer surface: the floating-cockpit composition that wraps the iframe with a topbar, slim sidebar, AnnotationsRail on the left, CanvasToolbar on the bottom, and Pins + PinLayer over the canvas. The iframe is full-bleed under the topbar — every chrome overlay is glass (--surface-glass-bg with blur(16px) saturate(140%)) and floats above the mockup. Implemented by src/components/MockupViewer/AppMainViewer.tsx + AppMainViewerWired.tsx; state lives in three focused hooks (useAppMainAnnotations, useViewerCanvas, useViewerFullscreen) and the iframe + PinLayer are extracted to a memoized ViewerCanvas so composer / rail churn doesn't reload the mockup.

React API — page composition (no Radix primitive for the surface)

The viewer surface is a custom compound — no Radix primitive backs the floating-cockpit layout. It composes the project's AppMain shell, the project-owned ViewerCanvas (iframe + PinLayer), AnnotationsRail, CanvasToolbar, VersionChip, AnnotationComposer, and MarkingBar sub-components. Cross-document anchoring (pins ↔ DOM nodes inside the same-origin iframe) goes through the runtime in docs/superpowers/specs/2026-05-18-pin-anchoring-strategy.md via useViewerCanvas. The version popover is the project's usePopover hook (custom, not Radix Popover). Confirmations for destructive thread actions go through useConfirm (Radix AlertDialog).

// src/components/MockupViewer/AppMainViewer.tsx — abridged
'use client';
import { AppMain } from '@/components/AppMain/AppMain';
import { AnnotationsRail } from '@/components/AnnotationsRail';
import { AnnotationCard } from '@/components/AnnotationCard';
import { AnnotationComposer } from '@/components/AnnotationComposer';
import { CanvasToolbar } from '@/components/CanvasToolbar';
import { MarkingBar } from '@/components/MarkingBar';
import { VersionChip } from '@/components/VersionChip';
import { ViewerCanvas } from './ViewerCanvas';        // memoized: iframe + PinLayer
import { useAppMainAnnotations } from './useAppMainAnnotations';
import { useViewerCanvas }       from './useViewerCanvas';
import { useViewerFullscreen }   from './useViewerFullscreen';

export function AppMainViewer(props) {
  const { annotations, activeId, setActiveId, marking, setMarking, pendingPins, …
        } = useAppMainAnnotations({ initialAnnotations, onCreate, … });
  const { iframeRef, canvasRootRef, iframeGen } = useViewerCanvas({ mockupSrc, onCanvasClick, marking });
  const { isFullscreen, toggle: onFullscreenToggle } = useViewerFullscreen(appMainRef);

  return (
    <AppMain ref={appMainRef} variant="viewer">
      <ViewerCanvas
        mockupSrc={mockupSrc} iframeRef={iframeRef} canvasRootRef={canvasRootRef}
        pins={pins} pendingPins={pendingPins} activeAnnotationId={activeId}
        onPinClick={setActiveId}
      />

      <AnnotationsRail badges={badges} activeId={activeId} onSelect={setActiveId}>
        {visibleAnnotations.map((a) => (
          <AnnotationCard key={a.id} {...a}
            active={a.id === activeId}
            onReply={…} onEdit={…} onDelete={…} onReact={…}
          />
        ))}
      </AnnotationsRail>

      <CanvasToolbar onFullscreen={onFullscreenToggle} fullscreen={isFullscreen} zoom={zoom} onZoom={…}>
        /* version popover slot — usePopover-anchored, custom (not Radix Popover) */
        <VersionChip versions={versions} current={currentVersion} onChange={…} />
      </CanvasToolbar>

      <AnnotationComposer open={composerOpen} pinCount={pendingPins.length} onPost={…} onCancel={…} />
      <MarkingBar open={marking} pinCount={pendingPins.length} onDone={() => setMarking(false)} />
    </AppMain>
  );
}

Notes:

  • No Radix primitive for the surface itself. Floating panels (rail, toolbar, composer, marking bar) are custom compounds with bespoke positioning, z-index, and motion. Radix is consulted only for confirmations (useConfirm → AlertDialog) and for the comment kebab menus (currently the project's usePopover, parked to move to @radix-ui/react-dropdown-menu in docs/future-features.md).
  • Three focused hooks. useAppMainAnnotations owns the list + reply/edit/delete/react state and the pending-pin/marking state. useViewerCanvas owns the iframe lifecycle + in-iframe click capture (per the pin-anchoring spec). useViewerFullscreen wraps the Fullscreen API + mirrors state for the toolbar button. This split is what makes ViewerCanvas memoizable.
  • ViewerCanvas is memoized. It receives only the iframe src + pin descriptors + active id; it does NOT re-render when the composer opens or the rail expands. Critical: without this, every composer keystroke would reload the iframe.
  • Wired vs unwired. AppMainViewer.tsx takes props for every callback (good for Storybook + tests). AppMainViewerWired.tsx binds it to live fetch against the API. The unwired component is also the one consumed by the (app) route via the wired wrapper.
  • Cross-document pins. The iframe is same-origin (mockups serve from /m/[mockupId]/[...path]); useViewerCanvas sets canvasRootRef to the iframe document's <html> after load. PinLayer renders pins at iframe-relative coordinates resolved by the anchoring runtime — see docs/superpowers/specs/2026-05-18-pin-anchoring-strategy.md.
  • Marking mode is state-driven. Setting marking = true changes the canvas cursor to crosshair via a CSS data-attr, mounts the MarkingBar, swaps the composer footer, and arms in-iframe click capture. Cancel clears pending pins; Done posts the annotation and exits marking mode.
  • Reply / edit / delete flow. Each card exposes its own kebab → action handlers from useAppMainAnnotations. Delete goes through useConfirm (Radix AlertDialog) before firing — see 14-dialog.
  • Client-side rendering. The page (src/app/(app)/projects/[slug]/[...path]/page.tsx mockup branch) is 'use client' per CLAUDE.md — payload is fetched in useEffect and forwarded to AppMainViewerWired.

Behavior
  • Same-origin iframe. Mockups serve from /m/[mockupId]/[...path]/route.ts on the same origin so the parent can reach into the iframe document for click capture and pin coordinate resolution. Cross-origin would break the anchoring runtime; the route handler is the gatekeeper.
  • Pin anchoring. Pins are stored as DOM-relative Anchor records (CSS selector + offset) — see src/lib/anchoring and the strategy doc docs/superpowers/specs/2026-05-18-pin-anchoring-strategy.md. On iframe load useViewerCanvas attaches the canvas root + resolves all anchors to absolute coords for PinLayer to render.
  • Marking mode lifecycle. Enter: composer opens or user hits ⌘⇧N. Canvas cursor → crosshair; MarkingBar mounts; in-iframe click handler armed. Click in iframe → drop pending pin with next color from palette. Done → POST annotation with pin anchors + body; pending pins promoted to real pins on success. Cancel → drop all pending pins, close composer, exit marking.
  • Active annotation. Clicking a canvas pin OR a rail card sets activeId — both render the matching pin glowing + the matching card with its .active treatment + (if rail was collapsed) the rail expands and stays pinned. Esc clears activeId.
  • Rail collapse / expand / pin. Hover → expand (300 px, motion-base spring). Pointer leave → collapse, UNLESS pinned. The lock pill toggles a persisted preference (per-user); pinned state survives mockup switches.
  • Replies / edit / delete. Each card's kebab opens an action menu. Delete goes through useConfirm (Radix AlertDialog) — destructive confirmation is non-skippable per 14-dialog. Reply opens an inline composer; Edit replaces the card body with a textarea.
  • Versions. The vchip opens the version popover (project's usePopover, not Radix). Selecting a version triggers an iframe src swap; useViewerCanvas re-runs anchor resolution against the new DOM. The agent loop's PATCH /api/mockups/[id]/version-patch creates new versions; the popover lists them with mono-typed names + author + time.
  • Fullscreen. The toolbar's fullscreen button + the F key both delegate to useViewerFullscreen which wraps the Fullscreen API on the AppMain container. Esc exits per browser default.
  • Drag affordances. Both the rail and the toolbar carry a grab handle (react-icons/go GoGrabber, 25 px). Rail rotates the glyph 90° to read horizontally; toolbar keeps vertical orientation. See 19-toolbar.
  • Color palette. 16 hues at curated chroma — propagates from data-color="0..15" on every pin / badge / dot-badge. Color is assigned by annotation order and survives across reloads (stored on the annotation row server-side). See 20-pins.
  • Agent loop entry point. This surface is the human counterpart of the automation agent's annotation-creation entry — see docs/agent-loop/INDEX.md. The same POST /api/mockups/[id]/annotations endpoint backs both the marking-mode "Done" submit here and the agent's drawing-based annotation flow.
  • Reduced motion. Rail width morph, pin pulse, pending-pin animation, composer enter — all collapse to 0 ms via the --motion-* token override at the top of the file. Functionality is unaffected.
