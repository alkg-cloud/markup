16 — Tooltip

Non-interactive hover hint. Single global <div popover="hint" id="markup-tooltip"> rendered once at the root layout (src/components/Tooltip/TooltipPortal.tsx); a capture-phase document listener watches mouseenter / focusin on every [data-tooltip] trigger, copies the text, positions against the trigger's getBoundingClientRect, and calls showPopover(). popover="hint" paints in the browser top-layer — escapes every overflow ancestor and stacking context, no z-index fight. ~150 ms dwell before show; instant hide on leave/scroll. Shares the glass-surface standard with dialog, popover, and rail/toolbar but with a smaller padding / radius / font footprint.

React API — custom popover="hint" (not Radix)

Backed by a custom in-project implementation rather than a Radix primitive. @radix-ui/react-tooltip exists but is NOT installed in this project — add it only when the first consumer needs Tooltip.Provider coordination (shared open-delay timer across multiple tooltips) or arrow-positioning. The current popover="hint" approach is simpler: one global element, one document listener, zero per-trigger React state.

// 1) Mount once in root layout
import { TooltipPortal } from '@/components/Tooltip/TooltipPortal';

export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html>
      <body>
        {children}
        <TooltipPortal />                              // single global <div popover="hint">
      </body>
    </html>
  );
}

// 2) Use anywhere — just attribute-driven, no component wrapper
<button
  data-tooltip="New annotation"                       // required — tooltip text
  data-tooltip-align="right"                          // optional: "left" (default) | "center" | "right"
  className={styles.kebab}
  aria-label="New annotation"                        // always pair with aria-label for SR

  <PlusIcon />

Notes:

  • @radix-ui/react-tooltip is not installed. Add to package.json only when the first consumer needs Provider-coordinated open-delays or an arrow. Until then, TooltipPortal is the canonical implementation.
  • Trigger contract is just an HTML attribute: data-tooltip="text" + optional data-tooltip-align. No React component wrapping required — works on any DOM element.
  • data-tooltip is not a replacement for aria-label. Pair them: data-tooltip drives the visible hint; aria-label drives the SR announcement. They may carry the same text, but the tooltip is aria-hidden at the element level (role="tooltip" on the portal alone is not enough for SR).
  • Tooltip content is pointer-events: none — never put interactive elements inside. For hover-triggered overlays with actions, use popover.
  • Only one global tooltip element exists; switching triggers reuses the same node (text + position get reassigned). This is what keeps the global capture-listener O(1).
State decision matrix
StateTriggerVisualAria
hidden Default. showPopover() not called. Nothing in the top-layer. The global element exists but is display: none. Element carries role="tooltip" but is unreachable by SR.
dwell mouseenter or focusin on a [data-tooltip] trigger. ~150 ms timer pending — nothing visible yet. —
shown Dwell elapsed. showPopover() invoked with text + position. Glass surface at calculated top/left, 6 px gap from trigger. Top-layer paint — never clipped. Element gets role="tooltip" + visible text; trigger's aria-label carries the SR announcement.
flipped (auto) Above-trigger placement would put top < 4 px. Switches to below trigger. [data-align] unchanged. —
dismissed mouseleave, focusout, scroll, resize, or another tooltip taking over. Instant hide — hidePopover() with no transition. Element returns to display: none. —
"dwell", "flipped", and "dismissed" are all internal states — the consumer never sets them. The portal owns the lifecycle entirely; consumers only paint data-tooltip on triggers.
When to use which
PrimitiveUse forBacking
tooltip Static hover/focus hint for icon-only or terse-label triggers (kebabs, zoom buttons, rail handles, version chips). No interactive content. Custom — popover="hint" + TooltipPortal
popover Anchored interactive surfaces — menu actions, status toggles, pickers. Click-driven, interactive content. See 15-popover.html
alert-banner Inline, non-dismissable contextual hints inside a form body (e.g. "you can't undo this"). Lives in the layout, not on hover. See 27-alert-banner.html
input-field hint Inline helper text under a form field. Always visible (or replaced by error). See 28-input-field.html
Rule of thumb: if the consumer must read it without acting, it's a tooltip. If they may act on it, it's a popover. If it lives next to a form field permanently, it's a hint.

Behavior
  • Single global element. One <div popover="hint" id="markup-tooltip"> mounted at root. Switching triggers reassigns text + position on the same node — no per-trigger React state.
  • Attribute-driven triggers. Any DOM element with data-tooltip="text" works. Optional data-tooltip-align="left|center|right" picks horizontal anchor.
  • Dwell delay. ~150 ms after mouseenter / focusin before show. Prevents flicker when sweeping the cursor across icons.
  • Instant hide. No exit transition on mouseleave / focusout / scroll / resize. Snappy beats smooth here — a delayed hide reads as "still hovering".
  • Top-layer paint. popover="hint" escapes every overflow ancestor and stacking context. No clipping by rail / toolbar / dialog overflow.
  • Position flip. Default placement is above the trigger; flips below when top < 4 px. Horizontal clamping keeps the tooltip inside the viewport with a 4 px gutter on both sides.
  • Focus-driven show. Tooltips appear on keyboard focus (focusin) — the hint is reachable without a mouse, which Biome's a11y rule requires.
  • Pointer-events: none. Tooltips never receive clicks. If you find yourself wanting interactive content, switch to popover.
  • Reduced motion. Under prefers-reduced-motion: reduce, the show animation collapses to a 0 ms opacity flip via the token override at the top of the file.
