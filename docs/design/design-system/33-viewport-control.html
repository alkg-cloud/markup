Viewport control

Lets a reviewer constrain the mockup iframe to a preset device size (Desktop 1440 × 900, Tablet 768 × 1024, Mobile 390 × 844) or a custom width × height. Composes a Radix Popover trigger that lives in the canvas-toolbar dock, three resize handles overlaid on the iframe in Custom mode, and sizing rules applied to the iframe inside mockup-view. Sibling popover surface: 15-popover.

React API — @radix-ui/react-popover + useViewport hook

The trigger and popover compose @radix-ui/react-popover (Root/Trigger/Portal/Content). Popover is installed in package.json and already powers FolderPicker. Viewport state and per-mockup persistence live in a custom useViewport(mockupId) hook that owns { mode, width, height, orientation } + a localStorage round-trip. Drag handles are a separate ViewportHandles subcomponent that reads the viewport state and emits resize events via the same setViewport setter.

// ViewportControl.tsx — composes Radix Popover + useViewport hook
import * as Popover from '@radix-ui/react-popover';
import { useViewport } from './useViewport';

<Popover.Root>
  <Popover.Trigger asChild>
    <button className="vp-trigger" aria-label={`Viewport: ${label(viewport)}`}>
      <ViewportIcon mode={viewport.mode} />
    </button>
  </Popover.Trigger>
  <Popover.Portal>
    <Popover.Content side="top" className="vp-popover">
      <ViewportPresetChips active={viewport.mode} onPick={setViewport} />
      {viewport.mode === 'custom' && (
        <ViewportSizeInputs width={viewport.width} height={viewport.height} onChange={setViewport} />
      )}
      <ViewportFooter mode={viewport.mode} width={viewport.width} height={viewport.height} />
    </Popover.Content>
  </Popover.Portal>
</Popover.Root>
  • Radix posture. @radix-ui/react-popover (Root/Trigger/Portal/Content) handles open-state, focus return on close, ESC/click-outside dismissal, and ARIA wiring (aria-haspopup, aria-expanded, aria-controls). side="top" opens upward because the toolbar lives at the canvas bottom.
  • Custom: preset chip radiogroup. The 5 chips are a custom role="radiogroup" with arrow-key navigation per WAI-ARIA — Radix does not ship a radiogroup primitive. Each chip is a <button role="radio" aria-checked>; the active chip carries [aria-checked="true"] which the CSS branches on for the tinted-active treatment.
  • Custom: useViewport(mockupId). Owns { mode, width, height, orientation }; persists to localStorage under viewport:${mockupId} (schema { v: 1, mode, width, height, orientation }). The hook also exposes the canonical preset table so callers don't hard-code sizes.
  • Custom: ViewportHandles. Only rendered when viewport.mode === 'custom'. Uses pointer events with setPointerCapture on pointerdown (same pattern as CanvasToolbar's drag-grab). Sets data-dragging="true" on the active handle for the visual state; Arrow keys nudge 8 px (Shift+Arrow 32 px) for keyboard accessibility.
  • Zoom coupling. Selecting a preset or Custom calls the parent's resetZoom() in the same tick. Implementation: the parent passes a handleViewportChange callback that wraps setViewport + resetZoom.
  • Pin reflow. Viewport width/height enters the existing repositionKey in AppMainViewer alongside zoom and layoutKey. PinLayer re-measures DOM anchors on key change, so pins ride along with the layout the mockup chooses for the new width.

When to use this vs canvas-toolbar zoom
Use viewport-control when…Use the zoom group when…
The reviewer wants to see how the mockup behaves at a specific device width (responsive validation, breakpoint review) The reviewer wants to inspect pixels at higher density — zoom is a CSS scale on the rendered output, not a layout change
The mockup's layout would differ at that width (media queries, container queries, JS-driven layout) Layout is fixed; only the rendering size needs to scale up or down for legibility
Pins need to be placed at locations that exist only at that breakpoint (e.g. a mobile-only CTA) Pin placement should not move based on viewport — preserve consistent anchors across reviewers

Behavior

  • Trigger click opens the popover (Radix-managed). ESC and outside-click dismiss. Focus returns to the trigger on close.
  • Preset / Custom selection resets zoom to 100% in the same tick — viewport and zoom are orthogonal in the spec but coupled in the parent dispatcher so the user never lands in a confusing "Mobile at 200% zoom" state.
  • Selecting Custom from a preset seeds W × H with the preset's current dimensions; the user adjusts from there.
  • Selecting Fit clears width/height to null and removes the iframe explicit sizing.
  • Rotate is only shown when Tablet or Mobile is active. Clicking it swaps width ↔ height (Tablet 768 × 1024 ↔ 1024 × 768).
  • Custom mode renders the drag handles overlaid on the iframe. Pointer events use setPointerCapture so a fast drag can leave the handle without losing the gesture. [data-dragging="true"] on the active handle persists the visual state until pointerup / pointercancel.
  • Custom min: 240 × 320. Drag clamps to canvas bounds; numeric input accepts above (excess becomes scrollable backdrop).
  • Drag handles are keyboard-accessible: focusable via Tab; Arrow keys nudge the dimension by 8 px; Shift+Arrow by 32 px. The right handle nudges horizontally; the bottom handle nudges vertically; the corner nudges both.
  • Per-mockup persistence via localStorage under viewport:${mockupId}. New mockups start in Fit; revisits restore the last chosen viewport. Storage failure (private mode / quota) falls back to in-memory for the session — no visible error.
  • Pin behaviour: viewport width/height enters the existing repositionKey; pins re-anchor when the mockup re-layouts. If a pin's anchor element is hidden at the new width, the existing pin-orphaned-anchor handling in useAppMainAnnotations kicks in.
  • Fullscreen: the chosen viewport persists into fullscreen. The toolbar remains visible; the backdrop fills around the constrained iframe.
  • Reduced motion: handle hover / active transitions and trigger transitions all use --motion-fast which the global media query zeros under prefers-reduced-motion: reduce.
