20 — Pins

Annotation marker on the mockup canvas. Custom in-project component (src/components/Pin/Pin.tsx) — no Radix primitive (Radix doesn't ship anything in this shape). 30 × 30 teardrop tags rotated -45° (border-radius: 50% 50% 50% 0) so the tip lands on the anchor point. Glass tag aesthetic: dark translucent bg, accent border tied to the annotation's colorIndex (0-15 palette), inner numeric label counter-rotated 45° to read horizontally. opacity: 0.55 at rest so the underlying mockup reads through; full opacity on hover / active. The 16-colour palette is shared with the rail badges (annotations-rail) and with author avatars elsewhere — the same colorIndex means the same hue everywhere a user encounters the annotation.

D — Anchor variants

Pins are anchored DOM-side and reposition on every layout change (resize, scroll, zoom, fullscreen). Two anchor shapes:

Element anchors store a CSS-path + fractional bbox offset (resilient to whitespace changes). Text anchors store a CSS-path + character offset + sub-position (resilient to reflow / wrap / font change). The pin itself is unaware — PinLayer writes the inline top / left.

React API — custom Pin + PinLayer (no Radix primitive)

Backed by two custom in-project components, no Radix primitive applies. <Pin> (src/components/Pin/Pin.tsx) is the stateless visual — receives colorIndex, label, status, onClick, and an optional ref that PinLayer uses to write inline top / left. <PinLayer> (src/components/PinLayer/PinLayer.tsx) owns the anchoring runtime: resolves each annotation's anchor against the live mockup DOM and reflows on every layout change.

// 1) Render a single pin — stateless presentation
import { Pin } from '@/components/Pin';

<Pin
  annotationId={a.id}
  colorIndex={a.colorIndex}                          // 0..15 — matches rail badge hue
  label={i + 1}                                      // "1", "2", ... — or "+" for pending
  status={a.id === activeId ? 'active' : 'idle'}
  onClick={(e) => { e.stopPropagation(); setActive(a.id); }}
  ref={el => pinRefs.set(a.id, el)}                  // PinLayer writes top/left here
/>

// 2) Mount the layer at the canvas — owns the anchoring runtime
import { PinLayer } from '@/components/PinLayer';

<PinLayer
  mockupRoot={mockupRef.current}                     // the mounted <iframe>.contentDocument root
  annotations={annotations}                          // each carries .anchor (element or text)
  activeId={activeId}
  pendingAnchor={composerAnchor}                     // renders the dashed pulsing pin during marking
  onPinClick={(id) => setActive(id)}
  zoom={zoom}                                        // reflows on zoom change
/>

// Anchor shapes (Pin doesn't see these — PinLayer does):
//   element: { path: string, offsetX: number, offsetY: number }      // 0..1 fractional
//   text:    { path: string, textOffset: number, subX: number, subY: number }
// PinLayer queries the path against mockupRoot, gets the bbox, applies offsets,
// writes pin.style.top / left. ResizeObserver + MutationObserver retrigger.

// Pins intentionally do NOT use tooltips — the rotated -45° transform makes

Notes:

  • No Radix primitive. Radix ships overlays / dialogs / menus — none of which fit a "30 px tag pinned to a sub-element of a sandboxed mockup". The teardrop shape, the colorIndex palette, the anchoring runtime, and the composer-pending state are all app-specific.
  • Pin is stateless presentation. All state lives in the parent: status, activeId, pendingAnchor. Pin renders the visual; PinLayer owns the layout.
  • colorIndex shared with rail badges. The same colorIndex on a pin and on the rail's .dot-badge means the same hue everywhere — that's how a user ties a pin on the canvas to a card on the rail visually.
  • Pin does NOT show a tooltip. The rotated -45° transform makes tooltip positioning flaky, and the rail card already labels the annotation (status + author + body). Pair aria-label with the pin instead — SR users hear "Annotation #001" on focus.
  • Anchor resolution lives in PinLayer. Two anchor shapes (element / text). ResizeObserver + MutationObserver + zoom-change watchers retrigger position writes. Pins are position: absolute within the layer.
  • Pending pin is composer-only. It renders during marking, before the annotation is persisted. Body is "+" rather than a number; border is dashed + pulses 1.6 s ease-in-out.

Behavior
  • Click to activate. Click any pin → annotation becomes active. onClick receives the synthetic event so callers can stopPropagation (default behaviour) before the canvas-level click handler dismisses focus.
  • Pin layer owns positioning. PinLayer resolves each annotation's anchor against the live mockup DOM and writes pin.style.top / left. ResizeObserver + MutationObserver + zoom-change watchers retrigger.
  • Anchor types. Element anchors store a CSS-path + fractional bbox offset; text anchors store a CSS-path + character offset + sub-position. Both reflow correctly on font / wrap / window-size changes.
  • Multi-pin annotations. One annotation can have multiple anchors → multiple pins with the same colorIndex + label. Activating one activates the annotation (all pins share the active visual).
  • Pending state lives for the draft's lifetime. While a DraftCard is mounted, every anchor in draft.pins renders as a dashed-outline pulsing pin on the canvas. Canvas click on empty area appends a draft pin; click on a draft pin removes it. On send, each pending pin transitions to the idle pin with the new annotation's number.
  • Resolved state is opacity-only. The consumer fades the pin to ~0.4 opacity when the annotation is marked resolved. Shape, hue, and label remain so users can still find the annotation by its mark.
  • No hover tooltip. The rotated -45° base transform makes tooltip positioning flaky. Pair the pin with aria-label for SR; the rail card carries the human-readable body.
  • Z-index hierarchy. Pins are z-index: 10 by default and 11 while active so the active pin reads above overlapping siblings.
  • Reduced motion. Under prefers-reduced-motion: reduce, the pin-pulse animation collapses to a 0 ms snap and the hover transition is instant — token override at the top of the file.
