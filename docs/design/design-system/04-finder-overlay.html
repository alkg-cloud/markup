04 — Finder Overlay

The "command palette" that opens from the finder-pill or via ⌘K/Ctrl+K. Ships at src/components/CommandPalette/CommandPalette.tsx and renders globally inside AppShell. Built as a custom compound — no Radix primitive backs this surface. Radix Primitives intentionally does not ship a Command/Combobox primitive; the cmdk library does, but the project rolls its own to keep zero runtime dependencies for search and to control the project-folder-mockup flattening + filtering pipeline (flatten.ts + filter.ts). Surface uses the glass standard (blur 16 px / saturate 140%, --border, --shadow-popover + --shadow-glow) shared with the rail, toolbar, and tooltips.

React API — custom compound (no Radix primitive)

Radix Primitives ships no Command / Combobox primitive — the surface is built end-to-end inside CommandPalette.tsx: state (open, query, selectedIndex), data pipeline (flattenProjectTree → filterAndGroup), keyboard handling, and next/router navigation. The component is mounted once in AppShell and listens for the global open-command-palette CustomEvent dispatched by the finder-pill.

import { CommandPalette } from '@/components/CommandPalette/CommandPalette';
import type { TreeProject } from '@/components/ProjectTree/ProjectTree';

function AppShell({ projects }: { projects: TreeProject[] }) {
  return (
    <>
      {/* … topbar, sidebar, main … */}
      <CommandPalette projects={projects} />       // mounts once, top-layer
    </>
  );
}

// Open from anywhere (no prop drilling) ─ same event the pill dispatches:
document.dispatchEvent(new CustomEvent('open-command-palette'));

Notes:

  • Single prop. projects: TreeProject[] — the full project tree. The component flattens it once via flattenProjectTree (memoised by reference) into a list of projects/folders/mockups searchable by name and path.
  • Open via CustomEvent (custom). The component listens for open-command-palette on document AND attaches a ⌘K/Ctrl+K listener. Both code paths set internal open = true. There is no open/onOpenChange prop because the trigger lives in a different subtree (the Topbar).
  • iframe shortcut rewiring (custom). A MutationObserver wires the keyboard listener into every same-origin iframe so ⌘K works inside the mockup viewer. The wrapper rewires on iframe.load because the mockup-serve route reloads the iframe contentDocument between version switches.
  • Keyboard contract. ↑/↓ move selectedIndex with wraparound; Enter activates the selected item via router.push(item.href); Escape closes; ⌘K/Ctrl+K toggles.
  • Highlight on match (custom). highlightMatch() wraps the query substring in <mark> and is rendered via dangerouslySetInnerHTML (the input is sanitised by escapeHtml() first).
  • Surface composition. .panel uses CSS Modules composes: floatingSurface from '../../styles/glass.module.css'. Same recipe as the dialog and popover — single source of truth for blur, border, and shadow.
  • Why not Radix Dialog? The palette has dialog-shaped behaviour (scrim, focus trap-ish, Escape closes) but Radix Dialog's API doesn't model a command-list with type-ahead. Using Dialog here would mean implementing the command list anyway and shoehorning the listbox semantics. Custom keeps the implementation closer to its data shape.

When to use which
SurfaceUse forBacking
finder-overlay Global search across projects, folders, and mockups. Type-ahead + keyboard navigation. Always reachable via ⌘K. Custom (this file)
dialog Forms (create / edit project, mockup), info overlays. Single primary action, no list semantics. @radix-ui/react-dialog
popover Anchored side-menus (kebab menus, tooltips with controls). Trigger-anchored, not centred. @radix-ui/react-popover
Rule of thumb: if the overlay is a search with results to navigate, use the finder. If it's a form with named fields, use Dialog. If it's a menu anchored to a trigger, use Popover.

Behavior
  • Open. CustomEvent open-command-palette or ⌘K/Ctrl+K sets open = true. The input is focused via useEffect after commit (RAF-based focus raced React 19's mount).
  • Type-ahead. 150 ms debounce on the input value; results re-grouped by section (Projects → Folders → Mockups).
  • Selection. Index resets to 0 every time the query changes so Enter on first keystroke always opens the top match. Pointer hover sets the index too.
  • Navigation. ↑/↓ wrap around the visible list; Enter calls router.push(item.href) and closes the palette.
  • Scroll into view. resultsRef.current.querySelector([data-index=N]).scrollIntoView({block:'nearest'}) on every selectedIndex change.
  • Close. Escape, scrim click, or successful navigation closes. No exit animation.
  • Iframe support. Listener attaches to every same-origin iframe document via a MutationObserver, rewiring on iframe.load so ⌘K works inside the mockup viewer even across version reloads.
  • Single overlay invariant. Opening the palette dispatches the same event the Topbar listens for to close the avatar menu — two overlays never stack.
  • No body scroll lock. Unlike dialog, the palette doesn't lock body scroll. The user can dismiss with Escape; if they manage to scroll the underlying page through the scrim that's not a correctness issue.
  • Reduced motion. Item enter animation and panel transitions collapse to 0 ms via the token override + a dedicated @media (prefers-reduced-motion: reduce) rule on .item.
